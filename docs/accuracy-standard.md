# 命理准确度评分标准

## 目标

把“这个产品到底准到什么程度”拆成可执行、可复跑、可解释的分数，而不是靠感觉说“差不多很准”。

## 三层口径

### 1. 盘面计算准确度

看的是：

- `lagna / moon / nakshatra / dasha` 是否和第二引擎对齐
- 是否存在 `major diff`
- 固定 benchmark case 的主字段是否稳定

这是最接近“排盘算得准不准”的分。

### 2. 解读可追溯性

看的是：

- 章节是否完整
- claim 是否足够
- `evidence_tag` 是否齐全
- 时间窗口、身份核验、remedy 是否齐全
- 当前大运是否进入报告上下文

这是“报告有没有纪律”的分，不是“人生事件已经被证明会发生”的分。

### 3. 实证验证成熟度

看的是：

- 有没有第二引擎交叉验证
- 有没有固定 benchmark case
- 有多少个 case
- 有多少专家复核
- 有多少长期回看样本

这是决定能不能对外说“已经很准”的关键分。

### 4. 反馈贴合度

看的是：

- 用户把具体 claim 标成 `准 / 半准 / 不准`
- `准` 记 1.0
- `半准` 记 0.5
- `有启发但不确定` 记 0.4
- `不准` 记 0

这是个人校准分，不等于总体命中率。样本少于 20 条时，只能用于改写报告，不可用于对外宣传准确率。

## 当前脚本实现

入口：

- `nadi_leaf/evaluation.py`
- `scripts/evaluate_reading.py`

输出字段：

- `quality_score`
- `accuracy_profile`
- `feedback_alignment`

## 95+ 的真实含义

### 可以说 95+ 的情况

- `盘面计算准确度 95+`
- `内部产品质量 benchmark 95+`

### 不能直接说 95+ 的情况

- “命理解读真实命中率 95+”
- “真实纳迪叶匹配准确率 95+”

除非：

1. 有足够多的 benchmark case
2. 有专家复核
3. 有长期回看样本
4. 多引擎仍保持低 diff

## 当前建议口径

1. 对内用：
   - `产品质量分`
   - `盘面计算准确度`
   - `实证验证成熟度`
   - `反馈贴合度`
2. 对外暂时只说：
   - `结构化吠陀盘面和纳迪风格解读已较稳定`
   - `产品仍处在研究增强阶段`
