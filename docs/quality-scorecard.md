# 质量评分框架

## 先把两个“分”分开

当前必须分成两套分数来看：

1. `产品质量分`
2. `可对外声称的命理准确度分`
3. `反馈贴合度分`

如果不拆开，这两个概念会被混淆，导致内部 benchmark 很高，但对外说法不诚实。

## 1. 产品质量分

这是当前已经在脚本里落地的 `score_product_quality(...)`，主要衡量：

- 盘面正确性
- 章节完整度
- 解释一致性
- 产品边界纪律
- 证据与评测准备度

这一分更像 `工程与产品完成度`。

当前 benchmark：

- 当前内部产品质量分：`98/100`
- 当前盘面计算准确度：`96/100`
- 当前解读可追溯性：`100/100`
- 当前实证验证成熟度：`46/100`
- 当前 accuracy profile：`82/100`
- 当前用户反馈贴合度：`80/100`，样本数 `5`
- 说明：它表示产品结构、输出纪律和内部验证链路已经比较稳，不表示“真实人生事件命中率 98%”

## 2. 可对外声称的命理准确度分

这是当前新增的 `score_accuracy_profile(...)`，专门回答“现在到底能不能说很准”。

它拆成三项：

### A. 盘面计算准确度（45% 权重）

看的是：

- 双引擎是否对齐
- 有没有 `major diff`
- 固定 benchmark case 主字段是否匹配

这项分高，意味着 `排盘和大运计算层` 已经相对可靠。

### B. 解读可追溯性（25% 权重）

看的是：

- 请求章节是否都输出
- claims 是否足够
- `evidence_tag` 是否覆盖
- 时间窗口、remedy、身份核验是否齐全
- 章节摘要是否还挂在当前大运语境里

这项分高，意味着 `报告写得稳`，不是意味着“人生预测全都命中”。

### C. 实证验证成熟度（30% 权重）

看的是：

- 有没有第二引擎交叉验证
- 有没有 benchmark case
- 有没有多 case 样例库
- 有没有专家复核
- 有没有长期回看样本

这项是最重要的刹车片。没有它，就不能把高内部质量分直接说成高准确率。

## 当前阶段真实结论

### 已达到 95+ 的部分

- `盘面计算准确度` 在当前 benchmark 上有机会达到 `95+`
- `产品质量分` 当前 benchmark 已达到 `98/100`

### 还没有达到 95+ 的部分

- `可对外声称的命理准确度分` 还不能诚实地说达到 `95+`
- 原因不是代码没跑通，而是：
  - benchmark case 还太少
  - 还没有专家复核
  - 还没有长期回看样本

## 什么时候才可以诚实地追 95+

至少要接近下面这个门槛：

1. `benchmark_case_count >= 20`
2. `expert_review_count >= 10`
3. `longitudinal_follow_up_count >= 10`
4. 双引擎交叉验证仍保持低 diff
5. 章节输出和 evidence tag 继续稳定

## 当前已知 blocker

1. `Rahu/Ketu` 度数仍有 minor diff。
2. 当前 benchmark 还是单样例主导。
3. 指纹分型仍是探索性模块，没有标注集。
4. 真实叶片语料和叶级标注仍缺失。

## 当前测试闭环

1. 单元测试
2. 固定出生样本回归
3. CLI demo 输出检查
4. 多引擎交叉验证
5. benchmark case 评分
6. 人工审阅 claim 与 evidence tag
7. 用户反馈 `准 / 半准 / 不准` 贴合度评分
